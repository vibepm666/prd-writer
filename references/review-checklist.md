# 模式 B：评审 Checklist

收到用户 PRD 后逐项检查，每项标注 **✅ / ❌ / ⚠️**。

## 方向层（概念版应覆盖）

- [ ] 明确的目标用户与核心痛点
- [ ] 清晰的一句话产品定位
- [ ] 明确的产品形态（Web / 小程序 / App / Skill）及选型理由
- [ ] 商业价值或变现逻辑（哪怕简单）
- [ ] 明确的「不做什么」边界
- [ ] 三视角（用户 / 商业 / 技术）均有基本答案

## 结构层（落地版骨架）

- [ ] 核心用户动线（最好有流程图）
- [ ] 结构化功能清单（带 🔴🟡⚪ 优先级）
- [ ] 关键页面布局线框图（导航、重点区域、元素位置）
- [ ] 每个 🔴 核心功能有独立 §5 详细描述（未合并省略）
- [ ] 功能清单与 §5 章节一一对应，无遗漏或幽灵功能

## 细节层（AI 写文档易偷懒处）

- [ ] 交互元素含全部状态（加载中 / 失败 / 禁用 / 空状态）
- [ ] 边界条件（空、超长、网络、权限、并发）
- [ ] 交互细节（反馈、危险确认、空状态引导、失败引导）
- [ ] 数据规范（字段、类型、长度、必填、默认值）
- [ ] 终端用户文案规范 + 风格基调
- [ ] 终端文案与开发字段说明分开
- [ ] 图表职责合理（复杂流程是否应迁出 diagram-generator）

## 评分参考

| 分数 | 含义 |
| --- | --- |
| 8–10 | 可直接进入开发/原型，仅少量待补充 |
| 5–7 | 方向清晰，结构或细节有明显缺口 |
| 1–4 | 方向或骨架未立住，建议先补概念版 |

输出结构见 `review-output-template.md`。
