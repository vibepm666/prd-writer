# 产品类型附录

写 **§7 非功能性需求** 及边界条件时，按产品类型 **触发式补充**（无关类型不强行展开）。

## 通用（所有类型默认写）

| 维度 | 要点 |
| --- | --- |
| 性能 | 首屏、核心操作响应时间 |
| 权限 | 登录门槛、角色/租户（如有） |
| 兼容性 | 目标设备、浏览器、系统版本 |
| 安全 | 传输 HTTPS、敏感字段脱敏展示 |
| 存储 | 单用户/全平台容量、保留策略 |

---

## B2B SaaS / 企业后台

**额外关注：**

- 多租户 / 组织架构 / 角色权限矩阵（管理员、成员、只读）
- 审计日志：谁、何时、做了什么（合规客户常要）
- 数据导出、批量操作、并发编辑冲突
- SLA 或可用性预期（若对客户承诺）
- SSO / 企业微信 / 钉钉登录（若提及）

**§7 示例补充**：「管理员可禁用成员账号；操作日志保留 180 天」

---

## To C 消费 / 内容 / 社区

**额外关注：**

- 内容审核、举报、封禁流程
- 推送 / 消息触达策略（若 App）
- 青少年模式、敏感内容（若 UGC）
- 支付与退款规则（若涉及交易）
- 分享裂变与隐私（微信场景）

---

## 营销站 / 落地页

**额外关注：**

- SEO 基础（标题、描述、结构化数据）— 需求级，不写实现
- 表单留资：字段、校验、提交成功/失败页
- 多语言 / 地区（若出海）
- 法务页：隐私政策、用户协议入口

**PRD 粒度**：通常 🔴 功能较少，§5 可聚焦首屏、定价、FAQ、留资表单。

---

## 内部工具 / 降本增效

**额外关注：**

- 使用人数与并发（内部常低估）
- 与现有系统对接范围（只记系统名与数据方向，不写接口）
- 权限：是否全员可用还是特定部门
- 迁移/导入历史数据（若有）

**商业视角**：价值表述为节省人力、减少错误、缩短流程，可弱化变现。

---

## 金融 / 医疗 / 未成年人 / 强合规

**触发条件**：用户提及支付、病历、实名、未成年、跨境数据等。

**额外关注：**

- 实名、资质、免责声明展示
- 敏感数据最小采集原则
- 操作留痕与可追溯
- 地域合规（个保法、GDPR 等）— 记 **产品侧必须满足的事实**，不写法律条文

信息不足时标 `[待补充：需法务/合规确认]`，不编造合规结论。

---

## AI 类产品

**额外关注：**

- 用户输入与模型输出的展示、编辑、重试
- 幻觉/错误时的免责与人工兜底
- 上下文长度与多轮对话边界
- 是否允许用户上传文件及类型限制

---

## 使用方式

1. 诊断阶段识别产品类型（可多选主类型）
2. 写 §7 时合并「通用 + 对应附录」
3. 写 §5 边界条件时，把类型相关项并入（如 B2B 的并发编辑、To C 的内容超长上限）

格式与粒度参考 `example-prd.md`。
