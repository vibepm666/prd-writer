# 三视角诊断与产品形态

在写任何文档之前完成。每次只问 1–2 个问题，像对话一样推进。

## 为什么要做

方向未对齐就写细节，容易写完才发现走不通。诊断用于提前对齐，再动笔。

## 视角 1：用户——需求真实存在吗？

- 目标用户现在怎么解决这个问题？用什么工具/方式？
- 现有方式哪里不够好？你的产品好在哪里？
- 是否跟真实用户聊过？他们如何描述痛点？

**目标**：确认需求真实存在，而非「我觉得用户需要」。

## 视角 2：商业——值得做吗？

- 怎么赚钱或对组织有什么价值？（订阅 / 广告 / 引流 / 降本）
- 市场有无类似产品？最大差异是什么？
- 目标用户规模？是否有足够空间？

**目标**：确认有可持续的商业逻辑。

## 视角 3：开发——做得出来吗？

- 核心技术能力是否具备？是否依赖第三方？
- 最难实现的部分是什么？
- MVP 必须有哪些功能？

**目标**：确认无技术死角，能找到可跑起来的最小版本。

## 产品形态选择

形态是用户接触产品的第一道门，诊断阶段就要想清楚。

**用户已说出形态**（如「做个 App」）：温和确认是原生 App、移动网页还是小程序，以及理由。

**用户未明确形态**：根据三视角信息 **主动推荐**，说明理由，不要只抛选项。

### 形态对比

| 形态 | 适合场景 | 核心优势 | 主要限制 |
| --- | --- | --- | --- |
| **网页 Web App** | 功能复杂、大屏、跨设备；B 端工具 | 开发快、跨平台、易分享、SEO | 移动体验弱、离线/推送受限 |
| **微信小程序** | 用户在中国、社交传播、中低频工具 | 无需下载、分享成本低 | 平台约束、不适合出海 |
| **原生 App** | 强设备能力、高频使用、明确变现 | 体验好、推送稳、可离线 | 成本高、审核慢、下载门槛 |
| **AI Skill / 插件** | 增强已有工具而非独立产品 | 成本低、借宿主用户 | 依赖平台、品牌难独立 |

### 推荐理由框架

1. 目标人群在哪里——用什么设备、从哪发现产品？
2. 核心功能——是否需要摄像头/GPS、离线、高频？
3. 团队约束——成本与上线速度？
4. MVP 优先——优先「最快验证需求」的形态，而非「最理想」

**多形态并存**：如「先网页验证，再出 App」，在概念版写清阶段策略。

## 诊断原则

- 三视角都要过，不必追求完美——基本清晰即可
- 某视角明显卡壳时，温和指出并一起想，不跳过
- 完成后口头总结产品方向，请用户确认，再写概念版
